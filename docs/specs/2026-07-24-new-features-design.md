## 0. Решения владельца (брейншторм + ревью, 2026-07-24)

- **Хранение:** «Локально + iCloud + мини-прокси». Полный бэкенд (Supabase) — не сейчас. Мультиюзер: каждый = свой
  телефон + свой iCloud, данные приватны.
- **Уровень сохранности:** **конфиг возвращается сам (iCloud Key-Value), история — файлом (экспорт/импорт в iCloud
  Drive).** Фото добавок в первой версии не восстанавливаются (переснять). CloudKit — не берём.
- **AI по добавкам:** максимальный автомат — карточка + рацион + напоминания; владелец подтверждает/правит. AI
  **извлекает инструкцию с упаковки**, но **не назначает дозу** (медицина → App Review).
- **AI-анализ результатов:** **отдельная последняя фаза**, вне первого релиза.
- **Прокси:** Cloudflare Worker + жёсткий лимит расходов. **Пре-сигнал звука:** дефолт 15 сек.

### ⚠️ Две фактические ошибки v1, исправленные в этой ревизии (проверено по коду)
1. **Финальный звук отдыха играет НЕ Live Activity, а локальное уведомление** (`mobile/utils/liveActivity.ts`
   `scheduleRestNotification()`: `sound: REST_SOUND`, `interruptionLevel: 'timeSensitive'`, DATE-триггер на `restEndsAt`).
   Live Activity только показывает состояние/таймер. → Второй сигнал делается тем же механизмом (см. §1).
2. **AsyncStorage по умолчанию ИСКЛЮЧЁН из iCloud-бэкапа** (`@react-native-async-storage/async-storage 2.2.0`,
   `RNCAsyncStorage.mm`: если ключ `RCTAsyncStorageExcludeFromBackup` не задан — дефолт `@YES`). → «Не потерять само
   собой» из v1 неверно. Сохранность строим явно (§3).

## ФИЧА 1. Двухступенчатый звук отдыха (громче + «скоро» / «начинай»)

### 1.1 Проблема
Один сигнал конца отдыха, приходит без предупреждения и субъективно слаб. Пользователь не успевает подойти к снаряду.

### 1.2 Как звук устроен сейчас (факты)
- **Foreground:** `hooks/useFeedback.ts` `playRestDone()` через `expo-audio` (`playsInSilentMode: true`) —
  **играет даже на беззвучном, пока приложение открыто.** Триггер в `app/workout-session.tsx` при `restRemaining <= 0`.
- **Locked / background:** финал — **локальное уведомление** (`utils/liveActivity.ts` `scheduleRestNotification`),
  планируется на `restEndsAt`; при старте отдыха кнопкой App Intent с локскрина планируется нативно
  (`targets/live-activity/_shared/CompleteSetIntent.swift`). Звук уведомления зависит от системных настроек
  (нужен включённый звонок; на аппаратном беззвучном не играет).

### 1.3 Решение — единый координатор двух сигналов
Оба сигнала — `rest-soon` (за `preSignalSeconds`, дефолт 15) и `rest-done` (в 0) — планируются **одним планировщиком
как локальные уведомления** с двумя детерминированными ID и полем `content.data.kind: 'rest_soon' | 'rest_done'`.
Единая операция `cancel/reschedule` при `startRest / addRestTime(+30) / pause / resume / endRest / finish`.
- **Foreground:** listener на доставку уведомления подавляет системный звук workout-события и проигрывает
  `expo-audio` (`playRestSoon()` / `playRestDone()`) + haptic — так сигнал **громкий и играет на беззвучном, пока
  экран открыт**. JS-порог по таймеру оставить только как fallback при отсутствии notification-разрешения.
- **Locked (App Intent старт отдыха):** нативная ветка `CompleteSetIntent.swift` обязана запланировать **ОБА**
  уведомления (RN спит и JS-порог не сработает). Сейчас планируется только финал — добавить `rest-soon`.
- **Звуковые файлы:** перегенерировать `rest-done.wav` громче/резче + добавить короткий отличающийся `rest-soon.wav`.
  Оба — bundled, короче 30 с, добавляются в config plugin и **требуют пересборки бинарника** (OTA-update ресурс не
  добавит).
- **Настройка:** `settings.preSignalSeconds` (дефолт 15, 0 = выкл) на экране Настройки, ключи i18n `ru/en/ua`.

### 1.4 Честные ограничения (в критерии, не обещать лишнего)
- **Громкий сигнал гарантирован в foreground** (экран отдыха открыт) — даже на беззвучном.
- **Locked + аппаратный беззвучный → звука нет** (iOS; обход только Critical Alerts, для таймеров Apple не
  одобряет — НЕ используем). На locked нужен включённый звонок. Убрать из acceptance формулировку «звучит на беззвучном».
- **Вибрация на locked** — только системная notification-haptic по настройкам пользователя; конкретный паттерн не гарантируем.

### 1.5 Критерии приёмки
На реальном iPhone: экран отдыха открыт → за ~15 с тихий пре-бип + лёгкая вибрация, в 0 громкий финал (оба на
беззвучном). Locked при включённом звонке → приходят два уведомления в нужные моменты, без задвоения. `+30 сек`,
`pause/resume`, ранний `endRest`, повторный тап App Intent — сигналы корректно перепланируются/отменяются. Смена
`preSignalSeconds` сдвигает пре-сигнал.

## ФИЧА 2. Добавки (спортпит): фото → AI → карточка + рацион + напоминания

Каркас раздела **уже есть** (`app/(tabs)/supplements.tsx`, `types/supplement.ts`, `data/supplementData.ts`,
`utils/supplements.ts`). Надстраиваем AI-ввод, готовое не ломаем.

### 2.0 Принципы раздела
- **Контент — в основном спортпит:** протеин, креатин, гейнеры, BCAA/аминокислоты, предтрен, витамины.
- **UI простой и понятный:** минимум шагов, крупные тапы, ясные слоты (Утро / Перед тренировкой / Вечер),
  расширенные поля (состав, продавец) — вторичны/свёрнуты.
- **Напоминания — центр раздела, дефолт ВКЛ.** Честная формулировка контракта — не «невозможно пропустить»
  (технически недостижимо: Focus, беззвучный, удаление, отзыв разрешения), а **«надёжно, best-effort + постоянно
  видимый статус защиты»**: если уведомления/звук выключены — заметный баннер «напоминания не сработают — включить».

### 2.1 Поток (UX)
1. «+ Добавить» → 2–3 фото (упаковка/состав) + поле «Своя заметка» (свободный текст: новая пачка, продавец X,
   ссылка где заказывать).
2. «Распознать» → сжатые фото + заметка → Cloudflare Worker (§4) → Gemini → структурированный JSON.
3. **Экран предпросмотра** — все поля редактируемые; для распознанных полей видно источник (этикетка / заметка /
   предположение). Ниже — предложенная схема приёма (редактируемая, **не выше дозы с этикетки**) и тоггл «Напоминания».
4. **Сохранить** → добавка в списке; напоминания планируются (§2.4). Постоянная копия фото — только после Save.

### 2.2 Модель данных (расширение `types/supplement.ts`, аддитивно, опционально)
```ts
type CompositionItem = { name: string; amount?: number; unit?: string; perServing?: boolean };
type ScheduleEntry = {
  kind: 'clock' | 'workout_start';
  time?: string;            // "HH:mm" для kind='clock'
  daysOfWeek?: number[];    // 1..7; пусто = ежедневно
  slot?: SupplementSlot;    // для группировки в UI
  enabled: boolean;
};
type Supplement = {
  // ...существующие поля без изменений...
  brand?: string;
  form?: 'powder' | 'capsule' | 'tablet' | 'liquid' | 'other';
  composition?: CompositionItem[];
  servingsPerContainer?: number;
  seller?: string;
  sourceUrl?: string;
  note?: string;
  photos?: string[];          // локальные URI (не восстанавливаются после переустановки — v1)
  schedule?: ScheduleEntry[]; // заменяет плоский reminderTimes
  createdBy?: 'manual' | 'ai';
};
```
- **Миграция аддитивная**, поля опциональны; стор версионирован (`STORAGE_KEY` v2 + `backfillSeedKeys`). Существующее
  плоское `schedule: SupplementSlot[]` сохраняем для совместимости UI; новые `ScheduleEntry[]` — для напоминаний.
- **Фото:** ImagePicker `quality` НЕ гарантирует размер → resize/transcode в bounded JPEG, проверка MIME/размера/
  ориентации, удаление EXIF/GPS, общий лимит запроса, очистка temp-файлов, копия в документную папку только после Save.

### 2.3 Новые зависимости
- `expo-image-picker` (SDK 54 — `~17.0.11`); обязательны строки разрешений camera/gallery.
- `expo-document-picker` (для импорта, §3).
- `expo-file-system` (проверить наличие). `expo-notifications` — уже есть.

### 2.4 Напоминания (надёжно, best-effort)
- **Агрегированные:** одно уведомление на **уникальное время** со списком «что + сколько» (напр. «08:00 — Витамин D3
  2000 МЕ, Омега-3 2 капс»). Резко снижает риск лимита iOS (~64 pending) и спам. `pre_workout` → событие старта
  тренировки, не часы.
- **Планирование:** repeating calendar trigger по `hour/minute` (wall-clock, БЕЗ фиксированной timezone). Не плодить
  one-shot на будущие дни. Хранить **декларативное расписание**, а не только notification IDs; при hydrate/foreground
  сверять с `getAllScheduledNotificationsAsync()`, чинить расхождения, показывать число активных напоминаний.
- **Бюджет 64** контролировать с учётом двух workout-сигналов; при приближении — предупредить.
- **Permissions:** единый pre-permission экран (объяснить зачем) до системного запроса — общий для workout и
  supplements, иначе первая тренировка заберёт системный prompt раньше добавок. Проверять не только `granted`, но
  `allowsSound` и lock-screen display; при `canAskAgain=false` вести в Настройки. Уважать `settings.notifications.supplements` (дефолт ВКЛ).
- **DST/timezone** — протестировать смену пояса и оба перехода DST.

### 2.5 Контракт Gemini (семантически безопасный)
- Ответ: `{ schemaVersion, status: 'ok' | 'needs_more_input' | 'unsupported', card, regimenDraft, warnings, missingFields }`.
- Каждое распознанное поле: `{ value, source: 'label' | 'user_note' | 'inferred', confidence }`; неизвестное — `null`,
  не догадка. При плохом/обрезанном фото → `status: 'needs_more_input'` с просьбой переснять конкретную сторону.
- **AI не назначает дозу/курс** — извлекает инструкцию производителя, предлагает редактируемый черновик не выше
  label-дозы. Лекарства/сомнительное → `unsupported` + ручной ввод. (Structured JSON не гарантирует истинность
  значений — предупреждает и Google.)
- **Валидация в два слоя:** Worker проверяет schema + бизнес-правила (лимиты длин/чисел/URL, допустимые единицы,
  `HH:mm`); клиент валидирует повторно; **ID генерирует клиент**, не модель.

### 2.6 Foreground notification handler
Текущий `Notifications.setNotificationHandler` (`utils/liveActivity.ts`) глушит звук/баннер для **любого**
уведомления → после добавления добавок он подавит их в открытом приложении. **Ветвить по `data.kind`**: workout-звук
подавляем (играем через expo-audio), supplement-уведомления показываем нормально.

### 2.7 Критерии приёмки
Сфотографировать реальную банку спортпита + заметка «купил у X» → карточка правдоподобно заполнена, видны источники
полей, схема не выше этикетки; правки работают; после Save добавка в списке, агрегированное напоминание приходит в
заданное время текстом «что + сколько»; плохое фото → просьба переснять, а не выдумка; фото сохранены и видны.

## ФИЧА 3. Сохранность данных (iCloud): конфиг сам + история файлом

### 3.1 Проблема
Всё в `AsyncStorage` (по умолчанию исключён из бэкапа, §0). Нужно: настройки/добавки возвращаются сами; историю не
потерять.

### 3.2 Решение (уровень выбран владельцем)
- **Конфиг — авто через iCloud Key-Value (committed, не optional).** Синхронизируем настройки + определения добавок +
  расписание (`ScheduleEntry[]`). Механизм — **короткий локальный Expo native-модуль поверх `NSUbiquitousKeyValueStore`**
  по образцу уже работающего модуля Live Activity (НЕ полагаемся на неподтверждённые сторонние wrappers). Учесть:
  entitlement iCloud KV, отсутствие iCloud-аккаунта, асинхронный первичный sync, `didChangeExternally`, конфликт
  local↔cloud (`schemaVersion` + `updatedAt`, last-write-wins), tombstones удаления, quota. Лимиты Apple: 1 МБ / 1024
  ключа — конфиг влезает с запасом; **историю в KV НЕ кладём.**
- **История — ручной экспорт/импорт в iCloud Drive.** Экспорт поверх существующего `utils/exportData.ts` (через
  share-sheet в Файлы/iCloud Drive). **Безопасный импорт** (новый, `expo-document-picker`): `schemaVersion`,
  строгая runtime-валидация, лимит размера, whitelist полей, миграции, preview числа записей, **авто-бэкап текущих
  данных перед заменой**, атомарное применение, понятная ошибка без частичного восстановления. Экспорт/импорт — только
  после `hasHydrated`. После импорта — **заново планировать supplement-уведомления**, не импортировать старые IDs.
- **Device backup (бонус):** опционально выставить `RCTAsyncStorageExcludeFromBackup = false` в Info.plist — тогда при
  ПОЛНОМ восстановлении телефона из iCloud данные тоже вернутся. Не заменяет KV/экспорт (обычная переустановка
  контейнер стирает).
- **Фото добавок** JSON-историей не восстанавливаются (URI мертвы после переустановки) — v1 принимает «переснять».

### 3.3 AI-анализ — ОТДЕЛЬНАЯ ПОСЛЕДНЯЯ ФАЗА (вне первого релиза)
По кнопке (История): агрегированная сводка (не сырые логи) → Worker `/analyze` → Gemini → текст-инсайты. Выносится за
рамки первого релиза (не влияет на сохранность/напоминания, расширяет privacy/App-Review-поверхность).

### 3.4 Критерии приёмки
Переустановить приложение → настройки и список добавок вернулись сами (iCloud KV); история восстановлена импортом из
файла в iCloud Drive; импорт битого/чужого файла — понятная ошибка без порчи текущих данных; после импорта напоминания
перепланированы.

## 4. Cloudflare Worker — мини-прокси (держатель ключа) + защита кошелька

### 4.1 Зачем
Единственная серверная часть. Держит `GEMINI_API_KEY` секретом (принцип AsbestosGuard). ~1 файл, бесплатный тариф.

### 4.2 Структура и защита
- `worker/` в репо, `wrangler.toml`, `src/index.ts`; секрет `wrangler secret put GEMINI_API_KEY`. **Отдельный Gemini
  project** под это.
- Эндпоинты: `POST /supplement-scan` (фото base64 + заметка → `{card, regimenDraft, ...}`), `POST /analyze` (фаза
  позже). Фиксированные model/prompt, **запрет произвольного proxying**.
- **Честно о защите:** статический app-token и CORS — **НЕ аутентификация** (токен извлекается из IPA/трафика; CORS
  не защищает native-клиент). Токен оставляем только как дешёвый фильтр случайного мусора.
- **Реальная защита кошелька (обязательна):** жёсткие лимиты body/image/token, timeout, **spend cap на стороне Gemini
  project + Cloudflare AI Gateway spend limit**, глобальный + per-IP fallback rate-limit (IP не считать личностью —
  NAT/privacy relay), логирование 429/расхода.
- **App Attest** (`@expo/app-integrity`, SDK 54, **alpha**) — сильная защита для публичного релиза, но альфа → отдельный
  spike, отложено. Пока (личное/узкий круг): токен + spend cap + rate-limit. Turnstile — не берём (в native требует WebView).
- **App Store privacy:** перед отправкой фото/заметки — экран-объяснение, что данные уходят Cloudflare и Google;
  задать retention, отключить лишнее логирование промптов/картинок. AI не позиционировать как медицинское назначение.

### 4.3 Конфиг приложения
Базовый URL Worker — в `app.json` → `extra` (читать `expo-constants`). Секретов в приложении нет.

## 5. Границы (out of scope)
- Полноценный бэкенд/БД, аккаунты, серверная синхронизация между устройствами, веб-дашборд, совместные фичи.
- Android-нативка. Critical Alerts. CloudKit. App Attest (только spike позже). Turnstile.
- **AI-анализ истории — не в первом релизе.** Воскрешение старых AI-фич прототипа — нет.

## 6. Риски iOS / SDK 54
- **Silent switch:** обычный notification-звук и `timeSensitive` не гарантируют звук на аппаратном беззвучном
  (foreground `expo-audio` — гарантирует). Acceptance «звучит на беззвучном» держим только для foreground.
- **JS заморожен на locked** → `haptics.light()` не сработает; только системная notification-haptic.
- **Лимит ~64 pending** уведомлений — контролировать; сверх — тихое отбрасывание дальних.
- **Reboot** scheduled-уведомления переживают; **uninstall/reinstall и сброс permissions — нет**.
- **Timezone/DST:** добавки — wall-clock (`hour/minute`); отдых — абсолютный `restEndsAt`.
- **Custom sounds** требуют пересборки бинарника (не OTA).
- **iCloud KVS:** сам API совместим с New Arch; риск — конкретный native-пакет. Без теста на реальном устройстве
  совместимость не утверждать → device spike перед внедрением.
- **ImagePicker:** обязательны permission-строки; multiple selection — только галерея, камера снимает последовательно.

## 7. Порядок реализации (каждая фаза — отдельный план)
1. **Звук** (Фича 1) — единый координатор двух уведомлений + нативная ветка App Intent (оба сигнала) + foreground
   fallback через expo-audio; перегенерация wav; настройка `preSignalSeconds`.
2. **Сохранность** (Фича 3.2) — безопасный экспорт/импорт (iCloud Drive) + iCloud KV native-модуль для конфига.
3. **Cloudflare Worker** (§4) — прокси + spend cap + лимиты, эндпоинт `/supplement-scan`.
